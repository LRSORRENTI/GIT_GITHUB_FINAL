REBASING, THE SCARIEST GIT COMMAND?

What really matters in this section:

Important:

  • Rebasing VS. Merging (Two ways of integrating changes from different bramches)
  • Git Rebase Basics
  • When NOT to rebase (Very important, often referred to as the 'GOLDEN RULE' of rebasing)

Why do we rebase?

Rebasing is often avoided by beginners because it can really mess things up, especially if you don't fully understand why/when you should rebase, and when you definitely SHOULD NOT use it

REBASING itself is an alternative to merging, and there are two main ways to use the git rebase command:

  • USE REBASING AS A CLEANUP TOOL
  • USE REBASING AS AN ALTERNATIVE TO MERGING

Comparing MERGING & REBASING

Let's look at the 'problem' with rebasing, let's pretend we're working on a project with 10 other developers, and we're all collaborating on a big codebase together

I start to work on a new feature, so I make a new feature branch, and I do some work on that feature branch, but a few hours later, one of my teammates finished the feature that they were working on, and that code is merged into main, so we want that work now, we usually never want to diverge from the main branch for a long time without gettingthose changes

So we then merge main into our feature and keep working, then we do more work, but the next day there's even more new changes on main branch, so we merge again, we merge master into our feature branch

So now our feature branch has a bunch of merge commits, if the main branch is very active, our feature branch history is muddied

This is something rebasing can help with

REBASING!

We can instead rebase the feature branch onto main branch, this moves the entire feature branch so that it BEGINS at the tip of main branch, all the work is still there, but we have re-written history

Instead of using a merge commit, rebasing rewrites history by creating new commits for each of the original feature branch commits

git switch feature
git rebase main

So when we rebase instead, we rewrite history, based upon the original feature branch commits, the tree becomes linear, feature branch is at the tip of main

Another way to rebase is to wait until we're fully done with our feature/code then rebase the feature onto main branch

Rebasing in essence

In essence rebasing is used when we want to clean up merge commit history, it creates brand new commits for each of our feature branch commits, no unnecessary merge commits, our project history is now linear

This won't screw up the main branch itself, but it will put all of our commits at the tip of the main branch, 'rebasing' them at the tip

This is demonstrated visually in GitKraken our GUI, before rebasing our commit history is non-linear, after using git rebase all our commits are in-line and on the tip of main

THE GOLDEN RULE, WHEN NOT TO REBASE

Never rebase commits that have been share with others, if you've already pushed commits to GitHub... DO NOT REBASE them unless you're positive no one on your team is using those commits

You can rebase your feature branch, but don't rebase other branches that are not yours.

Remember to rebase we use the git command from our feature branch, so the workflow looks like this :

  1. git switch feature
  2. git rebase main

When we use rebase we ALWAYS type the command from our branch, not anyone elses/main branch, always type git status, check and make sure you're on your own feature branch, then type git rebase main

Also when we rebase we can also end up with conflicts, in order to resolve, we can go into our conflicted file and we can manually remove them, or use VSCODE's input, that asks for accept both changes, accept incoming change.. etc, after we resolve them, back in the command line we can type: git rebase --continue

Cleaning up with INTERACTIVE REBASE

With interactive rebasing, we can reword commits, edit commits, 'fix up and squash' commits, drop commits, but all of this is nice to have, it's not extremely critical to the day to day git workflow

What matters in this section:

Nice to have:

  • Rewording commits
  • Fixing up / Squashing commits
  • Dropping commits

Sometimes we want to rewrite history, delete, rename, and even reorder commits (before sharing them) using git rebase!

Also remember, we don't want to rewrite others history, this is more for only our work, our branches. Cleaning up our history

The git rebase command for interactive rebase looks like: git rebase -i HEAD~4, this will enter into 'interactive mode', where we can edit commits, add files, drop commits etc.. Note we do need to specify how far back we want to go

Also keep in mind we're rebasing commits based on the HEAD of where they are based

Rewording commits with INTERACTIVE REBASE

Running git rebase with the '-i' flag will enter the interactive mode, which allows us to edit commits, add files, drop commits etc.. Note, we must specify how far back to go

Also know that we are NOT rebasing onto another branch, instead we're rebasing a series of commits onto the HEAD they're currently on

Okay so I just finished with colt's interactive rebase demo, and I got it to work, my default editor wasn't set, so I googled it and after running: git config --global core.editor "code --wait" that fixed the problem, I was then able to run: git rebase -i HEAD~9 and VSCODE automatically opened the visual based rebasing tool, easy to follow instructions, and git bash was waiting until I closed the interactive rebase, then changed that specified commit message

As a reminder, when VSCODE opens our interactive rebase, we have a few options to pick:

  • pick - use this commit
  • reword - use this commit, but edit the commit message
  • edit - use this commit, but stop for amending
  • fixup - use this commit contents but meld it into a previous commit and discard the commit message
  • drop - remove commit

Let's see how we can combine commits in our interactive rebase with VSCODE, to do this we can use the 'fixup' command

So if for example we have: pick 21acd17 add bootstrap, and: pick 4c6cb28 whoops forgot to add bootstrap js script, if we change pick 4c6cb28 to fixup 4c6cb28, the 4c6cb28 will disappear when we type git log --oneline, but it's still there, we can still see it on github, it just melded with the previous commit.

Seems a little confusing but the interactive rebase tool through VSCODE is really nice, it shows all the options and lays out how they work clearly

We can also completely remove commits with the interactive rebase option: 'drop', using drop will remove the commit entirely. Along with that commits contents.
